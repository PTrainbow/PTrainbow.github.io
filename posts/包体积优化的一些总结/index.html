前言

最近一直在搞包体积的优化，有点点心得

并且在优化的过程中发现了原有工程的各种问题，这里总结一下。

包体积优化的一些手段

这里先不讨论代码和依赖删减的部分，如果这些东西可以直接删除，那无疑是最快、最明显的方法

不过我也可以提供一个删减代码的经验：

  • 一定要理清业务流程和依赖，切断引用链即可，R8 自动会帮我们删减无用代码。
  • 少量多次的 commit，给自己一点容错空间~~

盲目的删减代码，只会带来无限的依赖报错，陷入泥潭。有时候可能只是一个方法的引用，把它注释掉就一切就都已经 OK 了。积累大量代码修改 commit 一次，后面出现问题想 revert 会很痛苦，因为有很多牵连的依赖问题。

再看回 Android APK 文件，APK 文件由多个部分组成

和包体积密切相关(体积比较大的)有几个部分：dex、native so lib、resource(asserts等)

那么优化的手段就是围绕着这三部分展开

dex

dex 就是我们的各种源码部分的体积。可以使用 apk analyzer classshark 具体问题具体分析

  1. kotlin
    如果你的工程中使用了 kotlin，那么你的 dex 无疑是会变大的。
    且同样的代码下，高本版 kotlin 本身占用更小体积。（1.6.x 和 1.36.x 差距几百K）
    同时，如果依赖了 kotlin-reflect 还会有 1M 左右的 class(不是 dex)额外体积。如果不需要的话，建议删除

  2. aar dependency
    可以通过 apk analyzer 分析 dex 中各个包名下的 class 体积分布，找到体积较大的部分。再通过 dependency 分析是由那部分引用到的？是否需要？从而做删减。

  3. proguard 规则
    release 一定要开启 minify（这个应该没人不知道吧···)
    没有特殊要求的话，可以打开 R8 optimize(proguard 中不要写 -dontoptimize)

    这里遇到过一个好玩的事情 —— R8 optimize 被卡主。当时有一个打点 model 类，这个类高达 30w 行代码，包含了所有的打点。R8 每次处理到这里就会死掉~~

    还有 keep 规则 问题，这个对于 dex 体积影响很明显。因为类名方法名参数等都被替换为极其简短的字母。keep 规则一定不要特别的宽泛！！！！，不要照抄别人的 proguard，使用 getdefaultproguardfile，在 proguard-rules 里添加自己的额外规则就好了

    这里也遇到过一个好玩的事情
    -keep @kotlin.Metadata class * { *; }
    这个规则直接 keep 住所有的 kotlin 代码，删除后，代码体积缩减20%

  4. R文件删除
    R文件在 dex 中也是有不少体积的，项目越大体积也越大，可以使用 booster 或者 bytex 删掉 R文件。
    R8 对于代码层面优化力度是挺大的，fullmode 没有尝试过，也不太确定 bytex 的其他一些包体积优化的插件是否有效(且 bytex 还为支持 agp7.0)，有兴趣可以自己试一下

  5. 似乎也有 dex 合并压缩的方案，没用过

resource

Android 资源不仅会增加 res 的大小，还会增大 resource.arsc 的大小，所以资源的删减有时候对于包体积的收益是很明显的。

  1. 常规的替换、压缩、删减资源(比如 png -> webp，asserts删减) 等等就不说了
    大部分图片转换为 webp 会带了不小的体积收益

  2. resguard
    目前看起来 R8 虽然也带有资源优化，但是远比不上 resguard 的功能。 resguard 会自动帮你剔除冗余资源，帮你对资源路径做混淆处理。会很明显的作用于 resource.arsc 的大小。
    而且最终使用 7zip 压缩，包体积进一步缩减。
    resource.arsc 还可以进一步压缩（API 30 以上被禁止了)
    但是，resguard 的 7zip 似乎对于要上 google play 的包有影响，而且并不支持 agp7.0，但是也有个人仓库对于 7.0 做了支持。

  3. resConfig
    如果没有过多要求，可以只保留 zh 和 xxhdpi 的资源目录

  4. aar 中的 resource
    特别注意如果工程中有类似 design-module(就是各种资源文件汇聚在一个的单独 module 甚至带了点代码)。可以尝试直接源码精简依赖，可能会有很大收益。

R文件的部分归结到 dex 中

native so

这部分没有特别多的办法，基本就是： 保留一种架构so 和 删减原有so。

当然，也有人为了进一步缩减包体积(对体积极其敏感的情况下)，将 多个so进一步打包压缩成一个小文件的方案，但是这样也会带来解压的问题，主要看个人需求。
