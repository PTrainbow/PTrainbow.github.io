dl_iterate_phdr 的 ANR 问题

最近一直在 hook 一些底层方法，写 c/c++。

可能目前写的 c/c++ 代码还会存在经不起推敲的情况，不过一些基本注意点已经知道了。

xhook/bhook 的作者是同一个人，bhook 有一个很重要的额外功能，就是可以自动监听后续加载的 so，通过 hook dlopen 来触发 refresh。

每次 refresh 后，都会使用 dl_iterate_phdr 来重新获取一遍所有的 so 的地址，然后执行替换。

问题就在这里，如果读取so、替换函数指针的过程，多次读取了 proc/self/maps，那么就会触发 ANR(bhook 有没有多次的读取，没有仔细看)。

目前工程有集成 bhook，但是我修改的 lib 中还是使用的 xhook，所以直接使用的 xhook + xdl(依然是 xhook 的作者)实现类似 bhook 的功能。

最终发现 dl_iterate_phdr 方法其实是一个危险的方法，会触发 ANR。

dl_iterate_phdr 带锁导致 ANR

起初 ANR 发生时，并没有往 dl_iterate_phdr 的锁问题上想。

因为 ANR 上报的信息中，显示主线程并没有持有任何锁，但是有大量的 IOW 状态。

而 xhook 在调用 elf_hook 的时候，总会重新读取 /proc/self/maps， 而 xdl 中，一开始会通过 dl_iterate_phdr 的 callback 执行到 elf_hook，那么众多 so 和 众多需要 hook 的方法，两者相乘就会触发成百上千次的 IO，那就可能会卡顿。

所以，以为是在主线程进行了 IO。可是查了半天发现整个 hook 的初始化过程都运行在子线程，子线程的 IO 怎么会直接 ANR 呢？难道是线程优先级太高，机器太垃圾？卡住了？将子线程的优先级降到最低后，依然 ANR，所以肯定不单单是 IO 的问题。

再后来，发现 dl_iterate_phdr 运行过程是 全局加锁的，所以说，只要 dl_iterate_phdr 没有执行完，想动用 linker，都会是一个阻塞的状态(不知道这里的表述是否正确，也不知道 dl_mutex 的范围是有多广，反正 activity 的跳转等等，是完全被阻塞住的)

int __loader_dl_iterate_phdr(int (*cb)(dl_phdr_info* info, size_t size, void* data), void* data) {
  ScopedPthreadMutexLocker locker(&g_dl_mutex);
  return do_dl_iterate_phdr(cb, data);
}

如何修复

那么其实已经知道了问题的所在，那么解决也就有对应的办法

首先，无论是移动到哪个线程去执行 dl_iterate_phdr 都是没用的

那么我们只能尽量 缩短 dl_iterate_phdr 方法的耗时

dl_iterate_phdr 在 xdl 的源码中，频繁的读取 /proc/self/maps(次数 = hook so 数量 * hook 的方法数)，如果我们可以 降低这个读取的频次，那就可以了！

所以，可以加一层缓存，当 so 加载完毕以后，只需要读取一次缓存即可。如果后续有 so 再加载，那么需要重新刷新一下缓存。

我们只需要在启动过程中，尽量靠后初始化 hook，让大部分的 hook 过程都依赖于第一次的缓存，那么整个启动过程就会很快，后续的一些懒加载的 so，只会各自触发一次缓存刷新，这个过程也很快，这样几乎 99% 解决问题。
